# DevDesk 个人开发学习工作台方案

## 1. 项目定位

DevDesk 是一款面向个人学习和开发实践的桌面端工作台应用。

它的目标不是一开始就做成完整的项目管理软件，而是作为一个长期学习项目：

- 记录自己正在开发的项目
- 管理学习任务和下一步计划
- 写学习日志和问题复盘
- 为后续开发的小应用预留入口
- 在持续迭代中学习桌面端、前端、数据存储和应用打包

一句话定义：

> 一个用于承载个人开发学习过程的桌面端工作台。

## 2. 使用场景

典型使用方式：

1. 新建一个学习项目，例如“学习 React”、“做一个番茄钟”、“练习 SQLite”。
2. 给项目记录当前目标、技术栈、本地路径和下一步任务。
3. 每天写一条学习日志，记录学了什么、遇到什么问题、怎么解决。
4. 在任务页查看今天要做的事情。
5. 后续开发的小工具，可以登记到 Apps 页面，作为入口统一管理。

## 3. 第一版风格

第一版采用简洁工作台风格。

设计原则：

- 界面清爽，不做复杂动画
- 先保证信息清楚、流程顺畅
- 页面数量少，但结构完整
- 不追求花哨视觉，优先学习真实开发流程
- 不做复杂插件系统，只预留未来扩展位置

## 4. MVP 功能范围

第一版 MVP 包含 6 个页面。

### 4.1 Dashboard 首页

展示个人学习和项目状态概览。

功能：

- 最近项目
- 今日任务
- 最近学习日志
- 简单统计：项目数量、任务数量、日志数量、已完成任务数量

### 4.2 Projects 项目管理

管理自己正在做或计划做的开发项目。

字段：

- 项目名称
- 项目状态：想法中、开发中、暂停、完成
- 技术栈标签
- 当前目标
- 下一步
- 本地路径
- 仓库地址
- 创建时间
- 更新时间

第一版操作：

- 新建项目
- 查看项目详情
- 编辑项目
- 删除项目

### 4.3 Logs 学习日志

记录每天的学习过程。

字段：

- 日期
- 关联项目
- 今天学了什么
- 遇到的问题
- 解决方案
- 明天继续做什么

第一版操作：

- 新建日志
- 查看日志列表
- 编辑日志
- 删除日志

### 4.4 Tasks 任务管理

管理项目中的具体待办事项。

字段：

- 任务标题
- 所属项目
- 状态：待做、进行中、完成
- 优先级：低、中、高
- 截止日期
- 备注

第一版操作：

- 新建任务
- 修改任务状态
- 编辑任务
- 删除任务

### 4.5 Apps 应用入口

作为后续小应用的入口登记页。

字段：

- 应用名称
- 应用类型：内置页面、本地路径、网页链接
- 描述
- 地址或路径

第一版先只做登记和打开入口，不做真正的插件系统。

### 4.6 Settings 设置

第一版只保留基础设置入口。

可先包含：

- 主题：浅色、深色、跟随系统
- 数据备份入口
- 数据导出入口

具体功能可以在后续版本再补。

## 5. 技术栈候选

目前主要考虑两个桌面端技术栈。

### 5.1 Tauri + React + TypeScript + SQLite

特点：

- 使用系统 WebView 显示界面
- 桌面能力由 Tauri 提供
- 后端能力通常通过 Rust 命令实现
- 应用体积通常更小
- 内存占用通常更低
- 安全模型更严格
- 需要接触一点 Rust 和 Tauri 配置

适合：

- 想做轻量桌面工具
- 想长期维护自己的个人软件
- 愿意逐步接触现代桌面端架构
- 可以接受前期环境配置稍微多一点

### 5.2 Electron + React + TypeScript + SQLite

特点：

- 内置 Chromium 和 Node.js
- 生态成熟，资料非常多
- 系统能力主要通过 Node.js 实现
- 开发体验更接近前端加 Node
- 应用体积通常更大
- 内存占用通常更高

适合：

- 想更快上手桌面端
- 希望资料多、问题容易搜索
- 想多练 JavaScript、TypeScript、Node.js
- 不太在意应用体积和内存占用

## 6. 移植性对比

这里的“移植”可以分成几类。

### 6.1 界面迁移

如果界面使用 React + TypeScript 编写，那么从 Tauri 迁移到 Electron，或者从 Electron 迁移到 Tauri，大部分界面代码可以复用。

可复用部分：

- 页面组件
- 表单组件
- 列表和详情页
- 路由结构
- 状态管理
- 样式代码

需要调整部分：

- 调用本地文件的代码
- 打开本地路径的代码
- 数据库访问方式
- 应用窗口和系统能力相关代码

结论：

> 只要把“界面逻辑”和“桌面能力”分开写，后期迁移成本可控。

### 6.2 数据迁移

SQLite 是本地数据库，适合个人工具，也方便迁移。

可移植优势：

- 数据存储在本地文件中
- 后续可以导出 JSON、CSV 或 Markdown
- 即使更换桌面框架，数据结构也可以继续使用

建议：

- 第一版就设计清楚 Projects、Logs、Tasks、Apps 的数据表
- 保留导出功能的规划
- 不把数据锁死在某个框架内部

### 6.3 桌面框架迁移

Electron 到 Tauri，或者 Tauri 到 Electron，迁移难度主要取决于本地能力封装得是否清楚。

建议做法：

- 前端页面只调用统一的服务层
- 服务层再去调用 Tauri 或 Electron 的本地能力
- 不在页面组件里直接写大量桌面 API 调用

例如：

```text
页面组件 -> projectService -> 本地数据库 / 桌面 API
```

这样后期如果换技术栈，主要改 service 层，而不是重写所有页面。

### 6.4 迁移到 Web 应用

如果后续想把 DevDesk 改成网页应用，也有一定可行性。

容易迁移：

- React 页面
- 大部分交互逻辑
- 表单和列表
- 项目、日志、任务的数据结构

需要重做：

- 本地文件路径能力
- 打开本地应用能力
- 本地 SQLite 访问方式
- 桌面窗口能力

结论：

> 它可以保留大部分前端界面，但桌面专属能力需要替换。

## 7. 当前推荐方案

考虑到当前目标是“通过开发软件学习桌面端开发”，并且前端基础还比较薄弱，推荐采用一个稳妥路线：

### 推荐路线：先 Electron，后 Tauri

原因：

- Electron 资料更多，上手更平滑
- 对前端基础薄弱更友好
- 可以把主要精力放在 React、TypeScript、组件、数据流和桌面应用结构上
- 等 DevDesk 跑通后，再用 Tauri 重做或迁移一版，会更容易理解两者差异

这不是否定 Tauri。Tauri 更适合作为第二阶段学习目标。

建议学习顺序：

1. 先用 Electron 做出可用 MVP。
2. 把界面和业务逻辑写得清楚。
3. 数据结构使用 SQLite，保持独立。
4. 后续新建 Tauri 分支或版本，尝试迁移。

这样可以同时获得：

- 更低的入门阻力
- 更完整的桌面端经验
- 后期比较两套技术栈的机会

## 8. 前端基础薄弱时的开发策略

第一版不要急着追求复杂架构。

建议采用以下策略：

### 8.1 先做静态界面

先不用数据库，不用复杂状态管理。

目标：

- 做出左侧导航
- 做出 Dashboard
- 做出 Projects 列表和详情
- 使用假数据展示页面

学习重点：

- HTML 结构
- CSS 布局
- React 组件
- props
- 基础事件处理

### 8.2 再做交互

加入表单和本地页面状态。

目标：

- 新建项目
- 编辑项目
- 删除项目
- 切换任务状态

学习重点：

- useState
- 表单输入
- 条件渲染
- 列表渲染
- 组件拆分

### 8.3 最后接入数据保存

等界面和交互清楚后，再接 SQLite。

目标：

- 数据能保存
- 关闭应用后重新打开还能看到
- Projects、Logs、Tasks、Apps 都接入真实数据

学习重点：

- 数据库表设计
- 增删改查
- 前端调用本地接口
- 错误处理

## 9. 阶段计划

### 阶段 0：准备和学习

目标：

- 明确项目范围
- 了解 Electron、React、TypeScript 的基本概念
- 搭好开发环境

产出：

- 项目方案文档
- 技术栈确认
- 简单页面草图

### 阶段 1：静态原型

目标：

- 完成应用主界面
- 使用假数据展示所有核心页面

功能：

- 左侧导航
- Dashboard
- Projects
- Logs
- Tasks
- Apps
- Settings

验收标准：

- 可以在桌面窗口中打开
- 可以切换页面
- 页面布局清楚
- 不要求真实保存数据

### 阶段 2：基础交互

目标：

- 页面可以进行基础操作
- 数据暂时存在前端内存中

功能：

- 新建、编辑、删除项目
- 新建、编辑、删除日志
- 新建、编辑、删除任务
- 新建、编辑、删除应用入口

验收标准：

- 不刷新时数据能变化
- 表单输入正常
- 操作流程顺畅

### 阶段 3：本地数据保存

目标：

- 接入 SQLite
- 实现真实本地保存

功能：

- Projects 数据保存
- Logs 数据保存
- Tasks 数据保存
- Apps 数据保存

验收标准：

- 关闭应用再打开，数据仍然存在
- 基础增删改查稳定

### 阶段 4：桌面能力

目标：

- 增加桌面端特有能力

功能：

- 打开项目本地路径
- 打开网页链接
- 打开应用入口
- 选择本地目录

验收标准：

- 可以从 DevDesk 跳转到本地项目目录
- Apps 页面可以打开登记的链接或路径

### 阶段 5：打包和复盘

目标：

- 打包成可安装或可运行的桌面应用
- 总结第一版学习收获

产出：

- 可运行的桌面应用
- 学习复盘文档
- 下一阶段计划

## 10. 版本路线

### v0.1 静态原型

- 桌面窗口
- 左侧导航
- 假数据页面

### v0.2 可交互版本

- 表单
- 新增、编辑、删除
- 前端内存数据

### v0.3 本地保存版本

- SQLite
- 基础数据持久化

### v0.4 工作台版本

- 打开本地路径
- 打开链接
- Apps 入口可用

### v1.0 自用稳定版

- 基础功能稳定
- 数据可备份
- 可以长期记录学习和项目

## 11. 暂不做的功能

为了控制第一版复杂度，以下功能暂时不做：

- 多用户登录
- 云同步
- 复杂插件系统
- 在线协作
- AI 自动总结
- 甘特图
- 完整看板拖拽
- 移动端应用

这些可以作为后续学习方向。

## 12. 初步结论

当前建议：

- 项目方向：个人开发学习工作台
- 第一版风格：简洁工作台
- MVP：Projects、Logs、Tasks、Apps、Dashboard、Settings
- 技术路线：优先 Electron + React + TypeScript + SQLite
- 长期路线：后续尝试迁移或重构为 Tauri 版本
- 开发节奏：先静态界面，再交互，最后数据保存和桌面能力

这个方案的核心价值不是一次做完一个大软件，而是让每个阶段都能学到明确的开发能力，并且每次学习成果都能沉淀回 DevDesk 本身。
